Цель: привести спецификацию в соответствие с реальным сценарием использования:

- один рекламодатель
- только чтение
- внутренний аналитический дэшборд
- собственное хранилище и визуализация

## Что лучше поменять

### 1. Название приложения

Текущее:

- `Morfeus Metrika Reporter`

Лучше сделать название, которое прямо связано с Direct и внутренней аналитикой:

- `Morfeus Direct Dashboard`
- `Morfeus Direct Analytics`
- `Morfeus Direct Spend Reporter`

Лучший вариант из этих трёх:

- `Morfeus Direct Analytics`

## 2. Назначение приложения

Лучше убрать акцент на локальный desktop-репортер и описать приложение как внутренний
аналитический сервис для одного рекламодателя.

Рекомендуемый текст:

```text
Приложение предназначено для автоматического получения статистики из Яндекс Директа
и построения внутреннего аналитического дэшборда по рекламным расходам и
эффективности кампаний.

Приложение используется одним рекламодателем для собственного аккаунта Яндекс
Директа и не предоставляет доступ третьим лицам.

Какие возможности даёт пользователям:
- автоматическая выгрузка данных по расходам, кликам, показам и конверсиям
- построение внутренних сводных таблиц и графиков по кампаниям
- сравнение показателей по периодам (неделя к неделе, месяц к месяцу, год к году)
- расчёт внутренних KPI, связанных с рекламными расходами
- использование данных исключительно в режиме чтения без изменения рекламных кампаний
```

## 3. Схема взаимодействия с API

Важно убрать формулировку, что всё происходит только локально на устройстве
пользователя, если по факту данные будут попадать во внутреннюю БД / дэшборд.

Рекомендуемый текст:

```text
Пользователь (владелец рекламного аккаунта или уполномоченный представитель)
авторизуется через OAuth 2.0 и предоставляет приложению токен доступа на чтение
данных Яндекс Директа.

Приложение отправляет запросы к API Яндекс Директа v5 для получения статистики
по рекламным кампаниям и связанным сущностям. Полученные данные загружаются во
внутреннее хранилище компании и используются для построения внутреннего
аналитического дэшборда.
```

## 4. Какие сервисы API указать

Лучше описать минимально необходимый набор.

Рекомендуемый набор:

- `Reports`
- `Campaigns`

Опционально, если планируете реально использовать:

- `AdGroups`
- `Ads`

Если они пока не нужны, лучше не раздувать заявку.

Рекомендуемая формулировка:

```text
Используются следующие сервисы API:
- Reports — получение статистики по кампаниям, расходам, кликам, показам и конверсиям
- Campaigns — получение списка и названий кампаний
```

## 5. Параметры отчётов

Этот раздел в целом нормальный, но лучше добавить, что отчёты используются не
только ежемесячно, а для регулярного обновления внутренней аналитики.

Рекомендуемый текст:

```text
Типичный запрос к Reports API включает:
- период: произвольный диапазон дат в рамках внутренней аналитики
- группировка: по кампаниям
- метрики: Cost, Clicks, Impressions, Ctr, AvgCpc, Conversions, CostPerConversion
- формат ответа: TSV
```

## 6. Обработка данных

Здесь лучше заменить акцент с "локально на устройстве пользователя" на "во внутренней
аналитической системе".

Рекомендуемый текст:

```text
Полученные данные обрабатываются во внутренней аналитической системе компании:
- агрегация расходов по кампаниям за выбранные периоды
- расчёт внутренних KPI и сравнений по периодам
- построение сводных таблиц и графиков для внутреннего дэшборда
- сопоставление рекламных расходов с внутренними бизнес-метриками компании
```

## 7. Безопасность и ограничения

Тут важно оставить максимально "безопасный" профиль:

- только чтение
- один рекламодатель
- внутреннее использование
- без передачи третьим сторонам

Рекомендуемый текст:

```text
- приложение работает только в режиме чтения и не изменяет рекламные кампании,
  объявления, ставки и другие объекты аккаунта
- приложение используется только для внутренней аналитики одного рекламодателя
- данные Яндекс Директа не публикуются в открытом доступе и не передаются третьим лицам
- доступ к данным и дэшборду ограничен уполномоченными сотрудниками компании
- OAuth-токен и технические параметры доступа хранятся во внутренней защищённой среде
```

## 8. Техническая реализация

Это один из самых важных разделов.

Текущее описание:

- `macOS (Claude Desktop + MCP)`

Лучше убрать это совсем. Для заявки это лишняя и потенциально запутывающая
деталь. Я бы писал нейтрально про внутренний сервис / аналитическую систему.

Рекомендуемый текст:

```text
Тип приложения:
Внутреннее аналитическое приложение

Платформа:
Внутренняя серверная аналитическая система компании

Версия API:
Яндекс Директ API v5

Авторизация:
OAuth 2.0 (Bearer token в заголовке Authorization)

Хранение токена:
Во внутренней защищённой конфигурации приложения

Нагрузка на API:
Низкая, регулярные запросы только для чтения статистики и обновления дэшборда
```

## 9. Частота запросов

Если вы правда строите дэшборд, лучше не писать `5–10 запросов в месяц`, если это
может не совпасть с реальной нагрузкой.

Безопасная формулировка:

```text
Частота запросов:
Низкая или умеренная, только для регулярного чтения статистики и обновления
внутреннего аналитического дэшборда
```

Если хочется конкретнее:

```text
Частота запросов:
До нескольких десятков запросов в сутки в режиме чтения
```

## 10. Что лучше не писать

Лучше убрать или не акцентировать:

- `Claude Desktop + MCP`
- "всё хранится только локально на устройстве пользователя", если это уже не так
- слишком узкий сценарий "ежемесячный отчёт", если вы строите живой дэшборд

- оно только читает данные Директа
- данные нужны для внутреннего аналитического дэшборда и отчётности
- доступ ограничен сотрудниками компании
- приложение не меняет кампании и не передаёт данные третьим лицам
