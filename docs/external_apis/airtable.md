```env

```

Важно:

- `.env.airtable` добавлен в `.gitignore`
- сами токены не хранить в markdown и не коммитить в репозиторий

## Как проверять доступ

Минимальная проверка токена:

```bash
curl -sS \
  -H "Authorization: Bearer $AIRTABLE_TOKEN" \
  "https://api.airtable.com/v0/meta/bases"
```

Если токен рабочий, API вернёт список доступных баз.

## Как смотреть структуру базы

Для метаданных по таблицам и полям:

```bash
curl -sS \
  -H "Authorization: Bearer $AIRTABLE_TOKEN" \
  "https://api.airtable.com/v0/meta/bases/$AIRTABLE_BASE_SMETOCHNAYA/tables"
```

Этот endpoint уже использовали и подтвердили, что он отдаёт:

- список таблиц
- поля
- типы полей
- views
- связи между таблицами

## Практическая заметка

Если нужно будет строить интеграцию дальше, логичный путь такой:

1. читать метаданные базы через `meta/bases/.../tables`
2. читать записи по нужным таблицам через обычные `v0/{baseId}/{tableName}` endpoints
3. складывать сырые выгрузки в raw-слой проекта
4. уже потом строить нормализованные витрины поверх них

## Что важно помнить

- для обычного чтения записей admin-права не обязательны; обычно достаточно доступа к базе и токена с нужными scopes
- токен, который уже попадал в чат, лучше считать засвеченным и перевыпустить
- Airtable API удобно использовать как источник и метаданных, и таблиц, но для управленческих дэшбордов лучше тащить нужные данные в нашу Postgres-базу
