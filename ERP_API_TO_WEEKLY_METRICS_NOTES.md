Практическая интерпретация:

- `total` — фактическая сумма продажи билета
- `nominal` — номинальная цена билета
- `discount_amount` — скидка
- `agent_id` — кандидат на канал продаж / агрегатор
- `seance_id` — ключ к джойну на сеанс

### Зафиксированное правило для ticket count

Для метрики "билеты в штуках" из ERP считаем только билеты, у которых:

- `total > 0`

Это оказалось самым близким приближением к weekly-метрикам количества билетов в базе.

Практический смысл:

- `all tickets` из ERP системно завышают count относительно weekly-базы
- фильтр `total > 0` убирает нулевые / бесплатные / служебные билеты и даёт почти совпадающий ряд

Значит, рабочее правило такое:

- если считаем ticket count из ERP для сверки с weekly-таблицей, берём только строки с `total > 0`
- это правило относится именно к **штукам билетов**, а не к денежной выручке

### Важный нюанс по `by-sell`

Сам endpoint `POST /tickets/by-sell` не всегда аккуратно режет выборку по `order_created`.
На практике корректнее:

1. брать более широкий диапазон;
2. дедуплицировать по `ID`;
3. уже у себя фильтровать по `order_created`.

Для weekly/monthly sales это важно.

### Как быстро ERP подтягивает продажи из Яндекс.Билетов

По наблюдению на свежих заказах ERP, похоже, подтягивает продажи из Яндекс.Билетов
батчами примерно раз в `10` минут.

Как это проверяли:

- брали недавние платные строки из `POST /tickets/by-sell`
- сравнивали `order_created` и `CreatedAt`
- агрегировали лаг только по нормальным заказам с ненулевым `order_id`

Что получилось на срезе за последние дни:

- Москва:
  - медианный лаг: около `6.0` минут
  - `p90`: около `9.8` минут
  - `p95`: около `10.1` минут
  - максимум среди нормальных кейсов: около `10.7` минут
- СПб:
  - медианный лаг: около `4.7` минут
  - `p90`: около `10.0` минут
  - `p95`: около `10.2` минут
  - максимум среди нормальных кейсов: около `10.5` минут

Дополнительный косвенный признак:

- `CreatedAt` часто попадает на минуты вида `:06-:08`, `:16-:18`, `:26-:28`,
  `:36-:38`, `:46-:48`, `:56-:58`

Рабочий вывод:

- для оперативных дашбордов ERP можно считать почти near-real-time источником
- но нормальный ожидаемый лаг между заказом в Яндекс.Билетах и появлением в ERP —
  это обычно `5-10` минут, а не секунды

## Названия спектаклей и `seance_id`

Названия спектаклей вытащить можно.

Логика:

- в билетах есть `seance_id`
- в `POST /shows/get` у сеансов есть:
  - `ID`
  - `event_title`
  - `show_start`
  - `show_end`
  - `venue_title`
  - `venue_city`
  - `hall_title`
  - `guests`
  - `tickets_count`
  - `cancelled`
  - `tickets`
  - `tickets_cert`
  - `tickets_invite`

Рабочий join:

```text
tickets.seance_id = shows.ID
```

Это даёт возможность:

- тянуть `event_title`
- строить продажи по спектаклям
- делить спектакли на типы (`Судный день`, обычные шоу и т.д.)
- считать гостей и загрузку по конкретным названиям спектаклей

### Зафиксированное бизнес-правило

- `СД` = `Судный день`

Значит, для ERP-derived метрик можно использовать простое правило:

- если `event_title = "Судный день"`, это `СД`
- иначе это не `СД`

## Что видно по сертификатам

Явный след сертификатов в ERP есть:

- `tickets_cert`

Это поле видно в данных по сеансам / шоу и позволяет отличать:

- обычные билеты
- билеты по сертификатам
- инвайты (`tickets_invite`)

Что это означает:

- ERP, похоже, умеет показать использование сертификатов на сеансах
- но в доступной коллекции пока не найден отдельный endpoint на продажу сертификатов как денежный поток

То есть:

- `tickets_cert` -> похоже на погашенные / использованные сертификаты
- `Выручка_сертификаты` -> в текущем виде скорее не из ERP напрямую, а из других источников / sheet / finance

### Что дополнительно выяснили про электронные сертификаты

Появилась сильная рабочая гипотеза, что продажи электронных сертификатов в ERP можно ловить
не по отдельному флагу билета, а по специальному `seance_id`, который соответствует
сертификатному мероприятию в Яндекс.Билетах.

Кейс, на котором это заметили:

- заказ `63004269`
- в ERP у него:
  - `seance_id = 40415759`
  - `sector = "Номинал на 2 персоны"`
  - `total = 980000` (то есть `9 800 ₽`)
- в интерфейсе Яндекс.Билетов этот же заказ показывается как:
  - `Подарочный сертификат (Москва)`

Что важно:

- `seance_id = 40415759` не нашёлся в ERP `shows/get`
- это отличает его от обычных спектакльных сеансов
- в ticket fields нет явного `is_certificate` / `certificate_id`
- `event_type` для сертификатов и обычных спектаклей одинаковый (`"1"`), поэтому
  для классификации он бесполезен

Из этого следует рабочее правило:

- `seance_id = 40415759` очень похоже на электронные сертификаты / сертификатное
  мероприятие

Проверка на weekly-данных:

- за неделю `06.04–12.04.2026` в ERP было `6` билетов с `seance_id = 40415759`
- их суммарный `total` составил `3 795 000` копеек = `37 950 ₽`
- в weekly-базе за ту же неделю метрика
  `Сертификаты (продажи) -> сертификаты электронные` = `36 432 ₽`

То есть ERP и weekly-база по этой линии оказались очень близки:

- ERP: `37 950 ₽`
- weekly `сертификаты электронные`: `36 432 ₽`
- дельта: `1 518 ₽` (~`4.2%`)

Практический вывод на текущий момент:

- `seance_id = 40415759` можно использовать как сильный маркер электронных
  сертификатов
- это пока именно рабочая гипотеза, а не окончательный справочник
- если позже найдём в ERP или Яндекс.Билетах справочник `event/seance -> activity`,
  этот mapping нужно будет формализовать

## Что из weekly metrics можно брать из ERP напрямую

Довольно уверенно:

- `Проведено шоу`
- `Проведено СД`
- `Гостей`
- `Среднее количество гостей`
- `% заполняемости`

Источники:

- зарплата на 1 СД

Для этого нужны:

- `salaries/period`
- `payments`
- `bonuses`
- `survey/satisfaction`
- `survey/answers/get-summary`

## Что ERP не покрывает как source of truth

ERP не выглядит хорошим первичным источником для:

- `Поступления на счет`
- банковских комиссий
- налогов
- маркетинга
- web-трафика
- CRM-пайплайна
- полного P&L
- франшизы / Китая / части B2B-финансов

Для этого нужны другие системы:

- PlanFact
- Яндекс.Метрика
- amoCRM
- Google Sheets как curated/fallback слой

## Самый важный вывод

ERP уже сейчас может быть source of truth для значительной части операционных weekly-метрик:

- спектакли
- гости
- загрузка
- билеты
- ticket sales
- спектакль-level разрезы
- часть актёрских и survey-метрик

При этом:

- денежные метрики уровня `Поступления на счет` и P&L лучше оставлять за PlanFact / finance layer
- derived weekly metrics лучше строить поверх ERP + business rules, а не только брать из Google Sheet
